# 三、系统设计

## 3.1 系统架构设计

本系统采用分层架构设计，将复杂的RAG流程分解为多个独立且可组合的模块层。整体架构从上到下分为用户界面层、应用逻辑层、检索处理层、知识表示层和数据存储层。

用户界面层基于Streamlit框架构建，提供Web交互式聊天界面。该层负责接收用户输入、展示系统回答、显示置信度分数和引用信息，以及提供系统配置选项。界面设计遵循简洁直观的原则，使用户能够方便地与系统交互并理解系统的工作状态。

应用逻辑层是系统的核心协调层，主要由MedicalChatEngine类实现。该层负责编排整个RAG流程，包括调用检索模块获取相关文档、调用重排序模块优化检索结果、计算置信度分数、执行安全路由决策、调用大语言模型生成回答，以及管理多轮对话历史。应用逻辑层通过清晰的接口与各下层模块交互，实现了流程的灵活控制和组件的松耦合。

检索处理层实现了混合检索和重排序功能。该层包含两个主要组件：HybridRetriever混合检索器和Reranker重排序器。混合检索器同时执行向量语义检索和BM25关键词检索，通过加权融合策略合并两路结果。重排序器使用交叉编码器模型对融合后的候选文档进行精排，提升最终检索质量。

知识表示层负责管理医疗知识库的索引结构。该层维护两个索引：FAISS向量索引用于语义检索，BM25索引用于关键词检索。向量索引存储文档的1024维嵌入表示，支持快速的相似度计算；BM25索引存储文档的词频统计信息，支持精确的关键词匹配。两个索引共同构成了系统的知识基础。

数据存储层管理原始医疗数据集和处理后的中间数据。系统使用Huatuo-26M和CMeQA等医疗问答数据集作为知识来源，通过DatasetLoader类实现数据的加载和预处理。动态分块器（DynamicChunker）将长文档切分为适合检索的文本块，并保留元数据信息用于引用溯源。

## 3.2 系统流程设计

系统的核心工作流程分为离线索引构建和在线问答两个阶段。

离线索引构建阶段首先加载医疗数据集，对数据进行清洗和格式化。然后使用BGE-M3嵌入模型将文本块转换为向量表示，构建FAISS向量索引。同时，系统对文本进行分词处理，构建BM25关键词索引。索引构建完成后持久化存储到磁盘，供在线查询使用。

在线问答阶段的工作流程如下：用户通过Web界面提交医疗问题，系统接收查询后首先进行查询预处理。然后并行执行两路检索：向量检索将查询编码为向量并在FAISS索引中搜索相似文档，BM25检索计算查询与文档的关键词匹配分数。两路检索结果通过加权融合策略合并，向量检索权重为0.7，BM25检索权重为0.3。融合后的候选文档经过BGE-Reranker重排序器精排，得到最终的检索结果。

系统计算检索结果的置信度分数，基于检索质量、上下文完整性和生成一致性等因素综合评估。根据置信度水平，系统执行安全路由决策：高置信度（≥0.7）直接生成回答，中等置信度（0.5-0.7）生成回答但附加警告，低置信度（<0.5）拒绝生成并建议咨询专业医疗人员。

对于中高置信度的查询，系统将检索到的文档构建为上下文，结合用户查询和对话历史，通过提示工程构造完整的提示词，调用Qwen3:8b大语言模型生成回答。生成过程中，系统要求模型基于检索到的上下文回答，使用引用标注格式标注信息来源，并在上下文不足时承认不确定性。

生成完成后，系统提取回答中的引用信息，格式化引用列表供用户查看。最终的回答、置信度分数、引用列表和路由决策结果一起返回给用户界面展示。

多轮对话管理贯穿整个流程，系统维护最近6轮对话历史，在生成回答时将其作为上下文信息传递给大语言模型，使模型能够理解对话的连贯性和上下文依赖。

## 3.3 核心模块设计

### 3.3.1 混合检索器（HybridRetriever）

混合检索器是系统的核心检索组件，实现了向量检索和BM25检索的融合策略。该模块接收查询字符串，分别调用向量检索和BM25检索获取候选文档，然后通过加权融合策略合并结果。

向量检索部分调用嵌入模型将查询编码为1024维向量，在FAISS索引中执行相似度搜索，返回Top-K个最相似的文档及其相似度分数。BM25检索部分对查询进行分词处理，计算查询与每个文档的BM25分数，返回Top-K个得分最高的文档。

融合策略采用加权求和的方式，对于同时出现在两路检索结果中的文档，将其向量检索分数和BM25分数分别乘以对应权重后相加，得到融合分数。对于仅出现在一路检索结果中的文档，仅计算该路的加权分数。最终按照融合分数对文档进行排序，返回Top-K个结果。

### 3.3.2 重排序器（Reranker）

重排序器使用BGE-Reranker-v2-m3交叉编码器模型对混合检索的结果进行精排。与双编码器模型不同，交叉编码器同时编码查询和文档，能够捕捉更细粒度的语义关系。

该模块接收查询字符串和候选文档列表，对每个查询-文档对计算相关性分数。交叉编码器将查询和文档拼接后输入Transformer模型，通过注意力机制捕捉两者之间的语义匹配程度。重排序后，系统按照新的相关性分数重新排列文档，返回Top-N个最相关的文档。

重排序显著提升了检索质量，特别是在处理医学术语的同义词、缩写和复杂表达时，交叉编码器能够更准确地判断语义相关性。

### 3.3.3 置信度评分器（ConfidenceScorer）

置信度评分器负责评估检索结果的质量，为安全路由提供决策依据。该模块综合考虑多个因素计算置信度分数：检索文档的平均相关性分数、检索结果的数量和分布、查询与检索结果的语义一致性等。

置信度分数的计算采用加权组合的方式，主要考虑检索质量（检索分数的均值和方差）和上下文完整性（检索结果是否充分覆盖查询的关键概念）。置信度分数归一化到0-1区间，用于后续的路由决策。

### 3.3.4 引用追踪器（CitationTracker）

引用追踪器负责记录和管理检索文档的来源信息，支持回答的引用溯源。该模块为每个检索文档分配唯一的引用编号，记录文档的来源、相关性和得分信息。

在生成回答时，系统将引用编号和文档信息传递给大语言模型，要求模型在回答中使用引用标注格式（如[1]、[2]）标注信息来源。生成完成后，系统提取回答中的引用标注，与原始引用信息关联，生成格式化的引用列表供用户查看。

### 3.3.5 聊天引擎（MedicalChatEngine）

聊天引擎是系统的主控模块，负责编排整个RAG流程。该模块协调检索、重排序、置信度评分、引用追踪和回答生成等各个步骤，实现完整的问答流程。

聊天引擎维护对话历史，支持多轮对话。在生成回答时，引擎将对话历史作为上下文信息传递给大语言模型，使模型能够理解对话的连贯性。引擎还实现了安全路由机制，根据置信度分数决定是否生成回答，在低置信度场景下拒绝回答并建议用户咨询专业医疗人员。

## 3.4 数据库与向量库设计

系统使用两种索引结构存储医疗知识：FAISS向量索引和BM25关键词索引。

FAISS向量索引存储文档的1024维嵌入表示。索引采用FlatL2距离度量，支持精确的相似度搜索。索引结构包含向量数据和元数据两部分，向量数据用于相似度计算，元数据存储文档的文本内容、来源信息和分块标识。索引支持批量添加和持久化存储，便于离线构建和在线加载。

BM25索引存储文档的词频统计信息。索引采用BM25Okapi算法实现，支持快速的关键词匹配查询。索引结构包含词汇表、文档频率和逆文档频率等信息，用于计算查询与文档的相关性分数。

两种索引共同构成了系统的混合检索基础，向量索引捕捉语义相似性，BM25索引捕捉关键词匹配，两者互补提升检索效果。

## 3.5 界面设计

系统基于Streamlit框架构建Web界面，采用简洁直观的聊天交互设计。界面主要分为三个区域：左侧边栏用于系统配置和状态显示，中间主区域用于聊天交互，右侧面板用于显示引用信息（可选）。

左侧边栏显示系统状态信息，包括当前运行模式（基础模式或SOTA模式）、检索配置参数和系统性能指标。用户可以通过侧边栏的控件调整检索权重、置信度阈值等参数，切换不同的处理模式。

中间主区域是聊天界面，采用标准的对话式交互设计。用户输入框位于底部，对话历史向上延伸。系统回答以消息气泡形式展示，包含回答内容、置信度分数和引用标注。置信度分数通过颜色编码直观展示：绿色表示高置信度，黄色表示中等置信度，红色表示低置信度。

引用信息在回答中以编号形式标注（如[1]、[2]），用户可以点击查看详细的引用列表。引用列表显示文档来源、相关性分数和文本片段，支持用户验证回答的准确性和可靠性。

界面设计遵循医疗应用的特殊性要求，在低置信度场景下显示明确的警告信息，提醒用户回答可能不可靠，建议咨询专业医疗人员。这种设计确保了系统在提供便利的同时，不会误导用户做出错误的医疗决策。
